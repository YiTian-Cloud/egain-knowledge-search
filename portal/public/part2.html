Search UI — UX & Frontend Architecture

This page summarizes the user experience and frontend design decisions behind the eGain Knowledge Search interface.

User Experience Highlights

  • Type-ahead search with real-time suggestions
  • Last 5 search history for quick reuse
  • Filtering by category and date
  • Sorting by relevance, date, and popularity
  • Search results with previews and metadata
  • Side panel article detail view
  • Save, open in new tab, and print actions
  • Responsive design for mobile and desktop

Frontend Architecture

  • React functional components + hooks
  • TypeScript-first development
  • Centralized reducer-based state management
  • Reusable, composable UI components
  • Mock backend abstraction for scalability
  • Personalization & theming (planned): User-aware preferences (role, locale, accessibility, theme) propagated via context to enable adaptive UI without fragmenting core components.

Project File Structure

The search UI is organized to separate UX components, state management, and backend abstractions, mirroring production-grade frontend architecture. 

src/
├─ api/
│  ├─ client.ts            # Shared HTTP client (base URL, headers, auth, error handling)
│  ├─ knowledgeApi.ts      # Knowledge search, suggestions, and article APIs
│  └─ metaApi.ts           # Analytics, metrics, and non-core supporting APIs
|
├─ pages/
│  └─ SearchPage.tsx        # Main search experience entry
│
├─ components/
│  └─ search/
│     ├─ SearchBar.tsx           # Query input + typeahead
│     ├─ SearchHistory.tsx       # Last 5 searches
│     ├─ SuggestionsList.tsx     # Suggestion list
│     ├─ FiltersBar.tsx          # Category/date sorting controls
│     ├─ FilterPanel.tsx         # Expandable filters UI
│     ├─ ResultsList.tsx         # Search results container
│     ├─ ArticleCard.tsx         # Individual result preview
│     └─ ArticleDetailPanel.tsx  # Side panel article viewer
│
├─ hooks/
│  ├─ useDebouncedValue.ts             # Input debouncingr
│
├─ state/
│  └─ searchStore.ts           # Centralized search state
│
├─ backend-pseudocode/
│  ├─ services/
│  │  └─ searchService.ts        # API abstraction layer
│  ├─ stores/
│  │  └─ articleStore.ts         # In-memory article store
│  └─ policies/                  # Ranking / filtering rules
│
├─ mocks/
│  └─ articles.ts                # Mock knowledge articles
│
└─ styles/
   └─ search.css                 # Lightweight styling

This structure allows the UI to evolve independently from the data source — enabling future migration to real APIs, caches, or AI services with minimal refactoring.

Extensibility

The architecture supports future enhancements such as image search, voice input, and AI-assisted answers — explored conceptually in Part 3.
